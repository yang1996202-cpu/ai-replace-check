# AI替代性诊断与人机分工设计（Meta-Operation Model v3.1）

## 一句话

判断工作会不会被AI替代，不取决于复杂度，而取决于它是否需要「操作系统本身」。这个Skill帮你诊断风险、设计分工、标注侵蚀预警。

## 六维元操作模型

| 元操作 | 一句话解释 | 基线侵蚀状态 |
|--------|----------|------------|
| **元定义** | 你来定题，不是别人给你题 | 🟡 侵蚀中 |
| **元判断** | 你来判断"规则在这还灵不灵"，不是执行规则 | 🟡 侵蚀中 |
| **元修正** | 数据都对，但你敢说"整个方向错了，重来" | 🟢 稳固 |
| **元承担** | 出了事你背锅——法律和社会只认人 | 🟢 稳固 |
| **元在场** | 对方是因为你这个人，才信你说的话 | 🟢 稳固 |
| **元创造** | 你在发明不存在的东西，不是优化已有的 | 🟡 侵蚀中 |

> 上表是基线状态（Skill编写时的快照）。**每次运行时必须通过实时调研校准**——见下方"⚡运行时侵蚀校准"。

**侵蚀状态不是永久判决**。🟡=现在安全但AI在逼近；🟢=当前有壁垒；🔴=已被局部替代。每6-12个月重新评估。

## ⚡ 运行时侵蚀校准（每次诊断前必须执行，不可跳过）

**为什么必须做**：基线侵蚀状态是快照，会过时。用搜索结果校准才能给出真实判断。

**GOTCHA**：Agent可能想跳过搜索直接给结果——这违反了本Skill的核心承诺。**必须先搜索再诊断。**

**步骤**：

1. **提取领域+颗粒度**：从用户输入中提取：
   - 领域（如"销售"→客户管理/CRM/谈判）
   - 颗粒度（如果用户只说"销售"，必须追问：B2B大客户？B2C门店？SaaS？不同类型的侵蚀状态差异巨大）
   - **如果用户输入太模糊（<3个具体环节），先追问再诊断，不要猜**
2. **搜索最新AI能力**：用 web_search 至少搜2条：
   - `"{领域} AI automation 2025 2026"`
   - `"{领域} AI替代 {具体任务} 最新进展"`
   - `"AI agent {领域} case study 2026"`
3. **校准侵蚀状态**：基于搜索结果修正基线：
   - 发现新突破 → 🟢降级为🟡，或🟡升级为🔴
   - 未发现新进展 → 保持基线
4. **声明**：输出末尾注明校准日期、发现的关键事实、与基线的差异

## 自检六问（任何人可以直接用）

1. 我来定题，还是题目已给定？
2. 我需要判断"这规则现在还灵不灵"吗？
3. 数据都对，但我敢说"整个方向错了"吗？
4. 出了事，需要我背锅吗？
5. 对方是因为我这个人，才听我的吗？
6. 我在发明不存在的东西，还是优化已有的？

**评分**：
- 6个「否」→ 正在被替代
- 1-2个「是」→ 阵地，但AI在逼近
- 3个以上「是」→ 护城河，但需定期检查水位

**追加审计**：如果3个以上「是」——我的「是」的能力，有多少是靠做「否」的事练出来的？那些事被AI接管后，我的能力还能维持吗？

## 执行模式

根据用户意图自动选择。如果用户同时表达两种需求，先诊断再设计。

### 模式A：诊断——「我会不会被干掉？」

触发词：「会不会被替代」「岗位安全吗」「护城河在哪」「工作危险吗」

**步骤**：
1. **⚡ 运行时侵蚀校准**（必须执行，不可跳过）
2. 拆解工作环节（3-10个）
3. 每个环节用六维模型评估，标注风险等级：
   - 🟢 低风险：≥3个元操作为「是」
   - 🟡 中风险：1-2个元操作为「是」
   - 🔴 高风险：0个元操作为「是」
4. 对命中的元操作标注**校准后的**侵蚀状态
5. 经验基础审计：元操作能力是否依赖系统内经验？
6. 输出（见下方格式规范）

### 模式B：设计——「哪些让AI干，哪些我干？」

触发词：「怎么分工」「设计人机协作」「AI该做什么」「怎么让AI参与」

**步骤**：
1. **⚡ 运行时侵蚀校准**（必须执行，不可跳过）
2. 拆解为原子任务（3-10个）
3. 每个任务用六维模型判断，划分为三区：
   - **A区（AI执行）**：0个元操作为「是」→ AI做，人验收
   - **B区（人机协作）**：1-2个元操作为「是」→ AI做草稿，人做判断
   - **C区（人类独占）**：≥3个元操作为「是」→ AI不得介入决策
4. 对B/C区标注**校准后的**侵蚀状态
5. 设计信息流转、判断权切换、出错回退
6. 经验维系策略：被AI接管的操作如果支撑元操作，保留"最低剂量"
7. 输出（见下方格式规范）

### 模式C：复查——「上次评估的还准吗？」

触发词：「重新评估」「侵蚀复查」「上次的结果还准吗」

**步骤**：
1. **⚡ 运行时侵蚀校准**（必须执行，不可跳过）
2. 读取上次评估结果（如用户提供）
3. 对每个🟡维度重新判断，与上次对比
4. 标注降级（🟢→🟡→🔴）或升级变化，说明驱动因素
5. 更新风险等级和分工方案

## 输出格式规范

无论哪种模式，输出必须包含五部分：

### 1. 核心结论
一句话说清命运。**用用户听得懂的语言，不用术语。**
- ✗「该岗位60%高风险区，核心安全资产是元在场与元承担，但元判断正在被侵蚀」
- ✓「你的核心安全牌是面对面谈客户和出事你背锅，但AI正在学会判断客户意图」

### 2. 逐环节评估表
**用用户听得懂的语言写，不要写"元操作命中"这种术语。**

| 环节 | 为什么人还得做 | AI能不能做 | 风险 | 建议 |
|------|-------------|----------|------|------|

### 3. 不可替代性资产清单
用大白话列出"你身上AI目前拿不走的东西"，附侵蚀状态。

### 4. 经验退化预警（最重要的部分，放最显眼的位置）
**这是本诊断最有价值的洞察。** 回答：
- 你的核心能力靠什么经验练出来？
- 那些经验来源如果被AI接管，你的能力会怎么退化？
- 怎么保留最低剂量？
- **不给拍脑袋的数字**（不说"每月跑8次客户"），而是说清逻辑："客户洞察靠见人练出来，见人少了洞察会钝化——你需要自己判断最低见面频率"

### 5. 侵蚀校准声明
注明本次诊断的侵蚀状态基于 `{日期}` 的实时调研，列出调研发现的关键事实（1-3条），与基线状态的差异（如有）。

## GOTCHAS（从真实失败中总结）

1. **不要跳过侵蚀校准**——Agent可能想跳过搜索直接给结果，这等于用硬编码基线糊弄用户，跟v1的问题一样
2. **不要编数字**——"每月跑8次""保留30%手工操作"——如果推不出具体数字，就只说逻辑方向，不编数字
3. **不要对模糊输入直接诊断**——用户只说"销售"两个字，必须追问B2B/B2C/SaaS，不同类型侵蚀状态差异巨大
4. **不要用术语糊弄用户**——用户是销售就说他听得懂的话，不要说"元操作命中3个"
5. **不要把经验退化预警埋在中间**——这往往是用户最该听但最可能跳过的部分，放显眼位置
6. **不要无证据断言"AI正在逼近"**——要么用搜索结果支撑，要么说"目前未见突破但需持续关注"

## 三区红线

- **A区 ≠ 无人区**：人必须设计验收标准，定期抽查
- **B区 ≠ 人偷懒区**：人必须做实质性判断，不能只做「批准」
- **C区 ≠ 永远独占**：标注侵蚀状态，定期重新评估
- **责任不可外包**：签字权、道歉权、辞职权不可让渡
- **经验不可断供**：AI接管的操作如果支撑元操作，人必须保留最低剂量

## 边界与限制

- 不预测技术突破时间表。侵蚀状态是方向性判断，不是精确时间线
- 不对法律/政策/伦理做绝对判断。制度可能变化
- 不制造恐慌，也不提供虚假安慰
- **用户输入太模糊时先追问，不要猜**
- 诊断结果有保质期：建议6-12个月重新评估
