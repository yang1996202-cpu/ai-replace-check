# AI替代性诊断与人机分工设计（Meta-Operation Model v3.2）

## 一句话

判断工作会不会被AI替代，不取决于复杂度，而取决于它是否需要「操作系统本身」。这个Skill帮你诊断风险、设计分工、标注侵蚀预警。

## 六维元操作模型

| 元操作 | 一句话解释 | 基线侵蚀状态 |
|--------|----------|------------|
| **元定义** | 你来定题，不是别人给你题 | 🟡 侵蚀中 |
| **元判断** | 你来判断"规则在这还灵不灵"，不是执行规则 | 🟡 侵蚀中 |
| **元修正** | 数据都对，但你敢说"整个方向错了，重来" | 🟢 稳固 |
| **元承担** | 出了事你背锅——法律和社会只认人 | 🟢 稳固 |
| **元在场** | 对方是因为你这个人，才信你说的话 | 🟢 稳固 |
| **元创造** | 你在发明不存在的东西，不是优化已有的 | 🟡 侵蚀中 |

> 上表是基线状态，来自高信源（Anthropic报告、MIT Autor&Thompson论文、专业任务平台的实际拆解数据）。**不是拍脑袋编的。**

**侵蚀状态不是永久判决**。🟡=现在安全但AI在逼近；🟢=当前有壁垒；🔴=已被局部替代。每6-12个月重新评估。

## 信息源分层体系（核心机制，决定了诊断的可信度）

**问题**：框架的权威性来自高信源，但开放web搜索会混入营销噪音、AI公司PR稿、未经证实的demo视频——用污水管给自来水补水，水质会下降。

**解决方案**：搜索是信号触发器，不是决策者。搜索只能拉响警报，不能直接修改侵蚀状态。

### 信源三层分级

| 层级 | 信源类型 | 可信度 | 能做什么 | 举例 |
|------|---------|-------|---------|------|
| **T1 确证层** | 同行评审论文、头部AI实验室技术报告、监管文件 | 高 | **可以改变侵蚀状态** | Anthropic技术报告、Nature/Science论文、欧盟AI法案、FDA审批文件 |
| **T2 信号层** | 主流科技媒体深度报道、知名投资机构行业报告、头部企业实际部署案例 | 中 | **可以触发复查，不能直接改状态** | The Information深度报道、a16z行业报告、"某某银行已部署AI做X"的真实案例 |
| **T3 噪音层** | AI公司博客/PR稿、社交媒体讨论、demo视频、未署名的"行业分析" | 低 | **只能作为搜索线索，不能作为判断依据** | "我们的AI能替代销售！"、Twitter上的hot take、知乎回答 |

### 校准规则（不可违反）

1. **只有T1信源可以改变侵蚀状态**（🟢→🟡、🟡→🔴）
2. **T2信源可以标注"⚠️ 待验证"**——在输出中告诉用户"有信号显示可能变化，但尚未被权威确认"
3. **T3信源不得影响任何输出**——可以用来发现搜索方向，但绝不能出现在诊断结论中
4. **如果没有找到T1/T2信源，保持基线状态，不要降级**——宁可保守，不要被营销忽悠

### 搜索时如何判断信源层级

- 来源是 `.edu` / `.gov` / 知名期刊 / 头部AI实验室官方 → T1
- 来源是 The Information / Reuters / Bloomberg / 知名投资机构 / 有署名有数据的企业案例 → T2
- 来源是 AI公司官网博客 / Medium / 知乎 / Twitter / 无署名报告 → T3

**GOTCHA**：AI公司说自己能做X，跟AI真的能在生产环境做X，是两回事。T3里充满了"we can"但缺"we did"。

## ⚡ 运行时侵蚀校准（每次诊断前执行）

**目的**：用当前AI能力的真实进展校准基线状态，但防止低信源污染框架。

**步骤**：

1. **提取领域+颗粒度**：从用户输入中提取：
   - 领域（如"销售"→客户管理/CRM/谈判）
   - 颗粒度（如果用户只说"销售"，必须追问：B2B大客户？B2C门店？SaaS？不同类型的侵蚀状态差异巨大）
   - **如果用户输入太模糊（<3个具体环节），先追问再诊断，不要猜**
2. **搜索最新AI能力**：用 web_search 搜索（至少2条）：
   - `"{领域} AI automation 2025 2026"`
   - `"{领域} AI替代 {具体任务} 最新进展"`
3. **信源分级**：对搜索结果按T1/T2/T3分级：
   - T1发现 → 直接校准侵蚀状态，在输出中注明来源
   - T2发现 → 标注"⚠️ 待验证"，保持基线状态但提醒用户关注
   - T3发现或无发现 → 保持基线状态，不降级
4. **声明**：输出末尾注明校准日期、信源层级、发现的关键事实

**核心原则**：
- **宁可保守，不被忽悠**——没有T1证据就不降级
- **搜索是雷达，不是方向盘**——它告诉你"可能有变化"，但改变方向需要高信源确认

## 自检六问（任何人可以直接用）

1. 我来定题，还是题目已给定？
2. 我需要判断"这规则现在还灵不灵"吗？
3. 数据都对，但我敢说"整个方向错了"吗？
4. 出了事，需要我背锅吗？
5. 对方是因为我这个人，才听我的吗？
6. 我在发明不存在的东西，还是优化已有的？

**评分**：
- 6个「否」→ 正在被替代
- 1-2个「是」→ 阵地，但AI在逼近
- 3个以上「是」→ 护城河，但需定期检查水位

**追加审计**：如果3个以上「是」——我的「是」的能力，有多少是靠做「否」的事练出来的？那些事被AI接管后，我的能力还能维持吗？

## 执行模式

根据用户意图自动选择。如果用户同时表达两种需求，先诊断再设计。

### 模式A：诊断——「我会不会被干掉？」

触发词：「会不会被替代」「岗位安全吗」「护城河在哪」「工作危险吗」

**步骤**：
1. **⚡ 运行时侵蚀校准**（必须执行）
2. 拆解工作环节（3-10个）
3. 每个环节用六维模型评估，标注风险等级：
   - 🟢 低风险：≥3个元操作为「是」
   - 🟡 中风险：1-2个元操作为「是」
   - 🔴 高风险：0个元操作为「是」
4. 对命中的元操作标注**校准后的**侵蚀状态（只有T1信源能改状态）
5. 经验基础审计：元操作能力是否依赖系统内经验？
6. 输出（见下方格式规范）

### 模式B：设计——「哪些让AI干，哪些我干？」

触发词：「怎么分工」「设计人机协作」「AI该做什么」「怎么让AI参与」

**步骤**：
1. **⚡ 运行时侵蚀校准**（必须执行）
2. 拆解为原子任务（3-10个）
3. 每个任务用六维模型判断，划分为三区：
   - **A区（AI执行）**：0个元操作为「是」→ AI做，人验收
   - **B区（人机协作）**：1-2个元操作为「是」→ AI做草稿，人做判断
   - **C区（人类独占）**：≥3个元操作为「是」→ AI不得介入决策
4. 对B/C区标注**校准后的**侵蚀状态
5. 设计信息流转、判断权切换、出错回退
6. 经验维系策略：被AI接管的操作如果支撑元操作，保留"最低剂量"
7. 输出（见下方格式规范）

### 模式C：复查——「上次评估的还准吗？」

触发词：「重新评估」「侵蚀复查」「上次的结果还准吗」

**步骤**：
1. **⚡ 运行时侵蚀校准**（必须执行）
2. 读取上次评估结果（如用户提供）
3. 对每个🟡维度重新判断，与上次对比
4. 标注降级（🟢→🟡→🔴）或升级变化，说明驱动因素和信源层级
5. 更新风险等级和分工方案

## 输出格式规范

无论哪种模式，输出必须包含五部分：

### 1. 核心结论
一句话说清命运。**用用户听得懂的语言，不用术语。**
- ✗「该岗位60%高风险区，核心安全资产是元在场与元承担，但元判断正在被侵蚀」
- ✓「你的核心安全牌是面对面谈客户和出事你背锅，但AI正在学会判断客户意图」

### 2. 逐环节评估表
**用用户听得懂的语言写，不要写"元操作命中"这种术语。**

| 环节 | 为什么人还得做 | AI能不能做 | 风险 | 建议 |
|------|-------------|----------|------|------|

### 3. 不可替代性资产清单
用大白话列出"你身上AI目前拿不走的东西"，附侵蚀状态。

### 4. 经验退化预警（最重要的部分，放最显眼的位置）
**这是本诊断最有价值的洞察。** 回答：
- 你的核心能力靠什么经验练出来？
- 那些经验来源如果被AI接管，你的能力会怎么退化？
- 怎么保留最低剂量？
- **不给拍脑袋的数字**，而是说清逻辑："客户洞察靠见人练出来，见人少了洞察会钝化——你需要自己判断最低见面频率"

### 5. 侵蚀校准声明
注明：本次诊断的侵蚀状态基于 `{日期}` 的调研；**列出信源层级**（T1/T2/T3）；发现的关键事实；与基线状态的差异（如有）；如果只有T2信号，标注"⚠️ 待验证"。

## GOTCHAS（从真实失败中总结）

1. **不要让T3信源污染诊断**——AI公司说自己能做X跟真的做了X是两回事，营销≠事实
2. **不要在没有T1证据时降级侵蚀状态**——宁可保守，不被忽悠
3. **不要编数字**——"每月跑8次""保留30%手工操作"——推不出具体数字就只说逻辑方向
4. **不要对模糊输入直接诊断**——用户只说"销售"两个字，必须追问B2B/B2C/SaaS
5. **不要用术语糊弄用户**——用户是销售就说他听得懂的话
6. **不要把经验退化预警埋在中间**——这往往是用户最该听但最可能跳过的部分
7. **不要跳过侵蚀校准**——虽然搜索结果可能全是T3，但"搜了没找到"和"没搜"是两回事

## 三区红线

- **A区 ≠ 无人区**：人必须设计验收标准，定期抽查
- **B区 ≠ 人偷懒区**：人必须做实质性判断，不能只做「批准」
- **C区 ≠ 永远独占**：标注侵蚀状态，定期重新评估
- **责任不可外包**：签字权、道歉权、辞职权不可让渡
- **经验不可断供**：AI接管的操作如果支撑元操作，人必须保留最低剂量

## 边界与限制

- 不预测技术突破时间表。侵蚀状态是方向性判断，不是精确时间线
- 不对法律/政策/伦理做绝对判断。制度可能变化
- 不制造恐慌，也不提供虚假安慰
- **用户输入太模糊时先追问，不要猜**
- 诊断结果有保质期：建议6-12个月重新评估
- **基线状态来自高信源，搜索只做信号触发——框架的权威性不来自搜索引擎**
