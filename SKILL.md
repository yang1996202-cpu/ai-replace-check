# AI替代性诊断与人机分工设计（Meta-Operation Model v3）

## 一句话

判断工作会不会被AI替代，不取决于复杂度，而取决于它是否需要「操作系统本身」。这个Skill帮你诊断风险、设计分工、标注侵蚀预警。

## 六维元操作模型

| 元操作 | 定义 | 基线侵蚀状态 |
|--------|------|------------|
| **元定义** | 定义问题边界——不是在给定框架内求解，而是决定框架本身 | 🟡 侵蚀中 |
| **元判断** | 判断规则是否适用——不是执行规则，而是判断"这个规则在此刻还成立吗" | 🟡 侵蚀中 |
| **元修正** | 修正整个框架——当异常暴露整个框架是错的时，敢推翻重来 | 🟢 稳固 |
| **元承担** | 承担决策后果——社会不认可非人主体的责任承担（制度性壁垒） | 🟢 稳固 |
| **元在场** | 在人际互动中作为「人」出现——信任建立在对等人格预设上 | 🟢 稳固 |
| **元创造** | 从虚无中提出新假设——不是组合优化，而是定义此前不存在的范畴 | 🟡 侵蚀中 |

> **重要**：上表是基线状态（Skill编写时的快照）。**每次运行时必须通过预诊断调研更新侵蚀状态**——见下方"运行时侵蚀校准"。

**侵蚀状态不是永久判决**。🟡表示"现在安全，AI在逼近"；🟢表示"当前有壁垒"（制度性或技术性）。每6-12个月重新评估。

## 自检六问

拿任何工作，问自己：

1. 元定义：我需要自己定义这道题的范围，还是范围已给定？
2. 元判断：我是否需要在执行中判断「规则不适用」？
3. 元修正：数据都指向一个方向，但我怀疑整个框架错了，我敢推翻重来吗？
4. 元承担：如果建议错了，需要我道歉/赔偿/辞职/被告吗？
5. 元在场：对方是因为「我」这个人而接受建议，还是因为建议本身合理？
6. 元创造：我是在优化已知流程，还是在发明从未存在的做法？

**评分**：
- 6个「否」→ 正在被替代
- 1-2个「是」→ 阵地，但AI在逼近
- 3个以上「是」→ 护城河，但需定期检查水位

**追加审计**：如果3个以上「是」——我的元操作能力有多少是靠做「否」的事积累出来的？系统内经验被AI接管后，我的元操作还能维持吗？

## ⚡ 运行时侵蚀校准（每次诊断前必须执行）

**问题**：Skill里的基线侵蚀状态是编写时的快照，会过时。解决方法是每次诊断前做实时调研，用当前AI能力的真实进展来校准侵蚀状态。

**步骤**（在拆解工作环节之前执行）：

1. **提取领域关键词**：从用户提供的岗位/任务中提取领域（如"销售"→客户管理/CRM/谈判；"审计"→底稿复核/风险识别）
2. **搜索最新AI能力**：用 web_search 搜索以下模式（至少2条）：
   - `"{领域} AI automation 2025 2026"` 
   - `"{领域} AI替代 {具体任务} 最新进展"`
   - `"AI agent {领域} case study 2026"`
3. **校准侵蚀状态**：基于搜索结果，对基线状态做修正：
   - 如果搜索发现AI在某个元操作维度有了新突破（如"AI销售谈判"已有落地案例）→ 🟢 降级为 🟡，或 🟡 升级为 🔴
   - 如果搜索未发现新进展 → 保持基线状态
4. **在输出中声明**：侵蚀状态基于 `{日期}` 的调研结果，基线版本为 v3（2026-06），校准依据附在诊断末尾

**关键原则**：宁可多搜一次，不要用过时信息给用户虚假安全感。

### 模式A：诊断——「我会不会被干掉？」

触发词：「会不会被替代」「岗位安全吗」「护城河在哪」「工作危险吗」

**步骤**：
1. **⚡ 运行时侵蚀校准**（见上方，每次必须执行）
2. 拆解工作环节（3-10个）
3. 每个环节用六维模型评估，标注风险等级：
   - 🟢 低风险：≥3个元操作为「是」
   - 🟡 中风险：1-2个元操作为「是」
   - 🔴 高风险：0个元操作为「是」
4. 对命中的元操作标注**校准后的**侵蚀状态（非基线状态）
5. 经验基础审计：元操作能力是否依赖系统内经验？依赖的话，系统内被AI接管后能否维持？
6. 输出（见下方格式规范）

### 模式B：设计——「哪些让AI干，哪些我干？」

触发词：「怎么分工」「设计人机协作」「AI该做什么」「怎么让AI参与」

**步骤**：
1. **⚡ 运行时侵蚀校准**（见上方，每次必须执行）
2. 拆解为原子任务（3-10个）
3. 每个任务用六维模型判断，划分为三区：
   - **A区（AI执行）**：0个元操作为「是」→ AI做，人验收
   - **B区（人机协作）**：1-2个元操作为「是」→ AI做草稿，人做判断
   - **C区（人类独占）**：≥3个元操作为「是」→ AI不得介入决策
4. 对B/C区标注**校准后的**侵蚀状态
5. 设计信息流转、判断权切换、出错回退
6. 经验维系策略：识别哪些被AI接管的操作支撑着元操作，保留"最低剂量"
7. 输出（见下方格式规范）

### 模式C：复查——「上次评估的还准吗？」

触发词：「重新评估」「侵蚀复查」「上次的结果还准吗」

**步骤**：
1. **⚡ 运行时侵蚀校准**（见上方，每次必须执行）
2. 读取上次评估结果（如用户提供）
3. 对每个🟡维度重新判断侵蚀状态，与上次对比
4. 标注降级（🟢→🟡→🔴）或升级变化，说明驱动因素
5. 更新风险等级和分工方案

## 输出格式规范

无论哪种模式，输出必须包含六部分：

### 1. 核心结论
一句话说清命运（如：「该岗位60%高风险区，核心安全资产是元在场与元承担，但元判断正在被侵蚀」）

### 2. 逐环节评估表
| 环节 | 元操作命中 | 风险等级 | 侵蚀状态 | 建议 |
|------|----------|---------|---------|------|

### 3. 不可替代性资产清单
只列出「人」能做到而AI目前做不到的具体能力，附侵蚀状态，用用户听得懂的语言。

### 4. 经验基础审计
元操作能力是否依赖系统内经验？如果依赖，给出维系策略。

### 5. 行动建议
具体到可执行的下一步（如：「客户分级报告交给AI，但保留每月两次亲自跑客户——你的元判断靠见人的直觉，不见人会钝化」）

### 6. 侵蚀校准声明
注明本次诊断的侵蚀状态基于 `{日期}` 的实时调研，列出调研发现的关键事实（1-3条），与基线状态的差异（如有）。

## 三区红线

- **A区 ≠ 无人区**：人必须设计验收标准，定期抽查
- **B区 ≠ 人偷懒区**：人必须做实质性判断，不能只做「批准」
- **C区 ≠ 永远独占**：标注侵蚀状态，定期重新评估
- **责任不可外包**：签字权、道歉权、辞职权不可让渡
- **经验不可断供**：AI接管的操作如果支撑元操作，人必须保留最低剂量

## 边界与限制

- 不预测技术突破时间表。侵蚀状态是方向性判断，不是精确时间线
- 不对法律/政策/伦理做绝对判断。制度可能变化
- 不制造恐慌，也不提供虚假安慰
- 用户不提供具体任务/JD时，先询问再执行
- 诊断结果有保质期：建议6-12个月重新评估
